## 概要

Optical Bead Computing（OBC）は、日本のそろばんの構造と操作論理に着想を得た、多値フォトニック情報処理のための提案フレームワークです。

そろばんでは、数値は単独の数字記号としてではなく、空間的な珠配置として表現されます。  
つまり、意味は線形な記号列ではなく、配置そのものによって担われます。

このフレームワークでは、同じ原理を光へ適用できる可能性を提案します。  
情報を、波長、偏光、位相、タイムビン、パルス幅、空間モード、軌道角運動量など、複数の光の自由度にまたがる**光珠パターン**として符号化し、その多次元光状態を変換・復号することで、計算または認識タスクを行うという考え方です。

このフレームワークは、次の三層を区別します。

1. 古典的な多値光符号化を用いる近未来の**決定論的レイヤー**
2. qudit研究の高次元状態概念を借用する**量子着想レイヤー**
3. 単一光子源と量子ゲートを含む長期的な**量子光学拡張**

このリポジトリでは、近未来の決定論的レイヤーを主な対象とします。

本ドキュメントは、概念モデル、状態形式、エラーの考慮、最小シミュレーションフレームワーク、プロトタイプ・ロードマップを示します。  
主張は、技術的に根拠を置ける範囲に限定します。

目的は、そろばん着想型フォトニック計算について、明確で再現可能かつ反証可能な研究フレームワークを提示することです。

## 概念的起源：そろばんから光珠へ

See also: [docs/soroban-origin.md](docs/soroban-origin.md)

### そろばん

そろばんは、桁ごとの軸に並ぶ珠の配置によって数を表します。  
各軸は位取りの1列に対応し、その列の珠配置が0から9までの1桁を表します。

そろばん上の完全な数値は、独立した数字列ではなく、全体の珠配置としての**空間パターン**です。

熟練したそろばん使用者は、数字を1桁ずつ逐次的に読むのではありません。  
全体のパターンを一度に認識し、操作します。

### フラッシュ暗算

フラッシュ暗算では、画面上に一瞬だけ表示される数列を見て、高精度に合計を計算します。

研究では、上級者は頭の中で仮想的なそろばんを動かしていると示唆されています。  
つまり、記号としての数字ではなく、空間的なパターンイメージを操作しているのです。

これは、数値操作が高速な**視覚的パターン認識と変換**として実行できることを示しています。  
この場合、パターンそのものが計算単位になります。

### そろばんから光珠へ

Optical Bead Computing は、この原理をフォトニック情報処理へ移します。

- そろばんが物理的な軸上の珠配置として数を符号化するように、OBQCは光の自由度上の多次元光配置として状態を符号化する。
- そろばん使用者が珠パターンを認識し変換するように、OBQCの復号器は光パターンを認識し、出力値へ対応づける。
- フラッシュ暗算が人間認知におけるパターン速度の利点を示すように、OBQCは光システムにおけるパターン型符号化の構造的利点を探索する。

中心となる対応関係は次の通りです。

| そろばん | Optical Bead Computing |
|---|---|
| 軸上の物理的な珠 | 光の自由度（λ, P, φ, …） |
| 珠の位置（上げる / 下げる） | その自由度内の状態値 |
| 全体の珠配置 | 光珠状態ベクトル B |
| そろばん計算 | 光状態変換 |
| フラッシュパターン認識 | 光パターン復号 |

## なぜ単なる二値フォトニクスではないのか

従来のデジタル電子回路や標準的な光通信システムは、情報レイヤーでは基本的に二値です。  
各チャネルは0または1を運びます。

高スループットな光ファイバーシステムでも、多くの場合、単一の光特性を切り替えることで情報を符号化します。  
たとえば、パルスの有無や、2状態間の位相差です。

OBQCは、これとは異なる構造を提案します。

| システム | 情報レイヤー | 1シンボルあたりの状態数 |
|---|---|---:|
| 二値電子回路 | 1本の配線につき1ビット | 2 |
| 従来型光通信 | オンオフキーイングまたはBPSK | 2 |
| 高度な光変調（QAM, OFDM） | 1シンボルに複数ビット | 4〜1024+ |
| **Optical Bead Computing** | **多次元の光珠状態ベクトル** | **利用可能な自由度によって設定可能** |

違いの本質は、単なる生の状態数ではありません。  
構造です。

OBQCは、多次元の光配置全体を、独立した二値チャネルの集合としてではなく、そろばんの珠配置に似た**単一のパターン単位**として扱うことを提案します。

この構造的な違いは、次の領域で利点を持つ可能性があります。

- 全体の状態ベクトルそのものが意味を持つパターン認識タスク
- 高次元クラスやテンプレートのコンパクトな符号化
- 複数の自由度を同時に使うことで、各チャネルの帯域要求を減らせる可能性がある光システム

ただし、これらの構造的利点が実際の計算上の利点に変わるかどうかは、物理実装可能性に依存します。  
これは未解決の研究課題です。

## 中核仮説

> 光の複数の自由度にまたがる識別可能な光状態を、信頼性をもって符号化・変換・復号できるなら、多値光珠計算は、二値光システムとは構造的に異なる、コンパクトで並列的かつパターン指向の情報処理を支援できる可能性がある。

この仮説は反証可能であり、実験的に検証できます。

重要な変数は次の通りです。

- 現実的なノイズ条件下で、各自由度ごとに信頼して識別できる状態数
- 複数自由度を結合符号化したときに達成できる総アルファベットサイズ
- 状態数、ノイズレベル、冗長符号化に対する復号精度
- パターン型符号化が二値符号化より構造的利点を持つ計算タスク

## 光珠状態モデル

See also: [docs/optical-bead-state-model.md](docs/optical-bead-state-model.md)

### 状態ベクトル

光珠状態は、次のタプルとして定義されます。

```text
B = (λ, P, φ, τ, w, s, ℓ)
````

各成分は、光の1つの自由度を表します。

| 記号 | 自由度           | 状態例                               |
| -- | ------------- | --------------------------------- |
| λ  | 波長 / 色チャネル    | 450 nm, 532 nm, 633 nm, 780 nm, … |
| P  | 偏光状態          | H, V, D, A, R, L                  |
| φ  | 位相状態          | 0, π/4, π/2, 3π/4, π, …           |
| τ  | タイムビン / 時間的位置 | bin 0, bin 1, bin 2, …            |
| w  | パルス幅          | narrow, medium, wide              |
| s  | 空間モード / 位置    | mode 0, mode 1, mode 2, …         |
| ℓ  | 軌道角運動量（OAM）   | ℓ = 0, ±1, ±2, …                  |

### 理論状態数と実用状態数

理論上の状態数は、各自由度の状態数の積です。

```text
N_theoretical = n_λ × n_P × n_φ × n_τ × n_w × n_s × n_ℓ
```

たとえば、4波長 × 4偏光 × 4位相 × 4タイムビン = 256理論状態です。

**ただし、実際に識別可能な状態数は、常に理論最大値より少なくなります。**

実用的な状態分離性は、次の要因に依存します。

* 検出器のスペクトル分解能と感度
* 信号対雑音比
* 各自由度における隣接状態間のクロストーク
* チャネルドリフトと環境不安定性
* キャリブレーション精度と安定性
* 自由度間の相互作用（例：偏光依存の位相シフト）

物理実装ごとに、実際に利用可能なアルファベットサイズは実験的に決定する必要があります。

このリポジトリのシミュレーターは、ガウスノイズモデル下での分離性を推定するためのツールを提供します。

## 決定論的OBQCと量子OBQC

See also: [docs/deterministic-vs-quantum.md](docs/deterministic-vs-quantum.md)

| レイヤー                            | 説明                                         | 状態                   |
| ------------------------------- | ------------------------------------------ | -------------------- |
| **決定論的 Optical Bead Computing** | 巨視的な光源と検出器を使う古典的な多値光状態符号化・復号               | 近未来シミュレーション / プロトタイプ |
| **量子着想型OBQC**                   | 完全な量子コヒーレンスや量子もつれを要求せず、qudit的な高次元符号化概念を用いる | 概念 / 研究フレームワーク       |
| **量子 Optical Bead Computing**   | 単一光子、qudit、量子もつれ、量子ゲートを用いる将来拡張             | 長期研究方向               |

決定論的レイヤーが近未来の主な対象です。

これは量子コヒーレンスを必要とせず、次のような構成で試作できます。

* ダイオードレーザーまたはLED
* 標準的な偏光光学素子
* 光変調器
* カラーセンサーまたは分光器
* Pythonベースの復号ソフトウェア

量子拡張は長期的な研究方向であり、決定論的プロトタイプの前提条件ではありません。

## 技術アーキテクチャ

```text
┌─────────────────────────────────────────────────────────────────────┐
│                   Optical Bead Computing Architecture               │
│                                                                     │
│  ┌──────────┐   ┌───────────────┐   ┌──────────────────────────┐   │
│  │ Encoder  │──▶│ Optical State │──▶│  Optical Bead Channel /  │   │
│  │          │   │  Generator    │   │  Transformation Layer    │   │
│  └──────────┘   └───────────────┘   └──────────────────────────┘   │
│                                                  │                  │
│  ┌──────────────────────────────────────────┐   │                  │
│  │  Error Correction / Redundancy Layer     │◀──┘                  │
│  └──────────────────────────────────────────┘                      │
│                          │                                          │
│             ┌────────────▼────────────┐                            │
│             │  Decoder / Pattern      │                            │
│             │  Recognition Unit       │                            │
│             └────────────────────────┘                            │
│                          │                                          │
│             ┌────────────▼────────────┐                            │
│             │   Software Simulator /  │                            │
│             │   Analysis Layer        │                            │
│             └────────────────────────┘                            │
└─────────────────────────────────────────────────────────────────────┘
```

構成要素の説明：

* **Encoder:** 入力値やデータを光珠状態ベクトルへ対応づける。アルファベット定義と状態割り当てを実装する。
* **Optical State Generator:** 指定された波長、偏光、位相、時間、空間特性を持つ光パルスを生成する物理またはシミュレーション上の光源。
* **Optical Bead Channel / Transformation Layer:** 光状態が通過する伝搬媒体または変換ネットワーク。ビームスプリッター、波長板、位相変調器、遅延線、空間光変調器などを含みうる。
* **Error Correction / Redundancy Layer:** シンボルエラー率を下げるため、冗長符号、パリティチェック、前方誤り訂正などを適用する。
* **Decoder / Pattern Recognition Unit:** 受信状態の光学特性を測定し、定義済みアルファベット内の最も近い状態へ対応づける。ハードウェアでは多チャネル検出器アレイと信号処理、シミュレーションでは最近傍距離計算に相当する。
* **Software Simulator / Analysis Layer:** 符号化、ノイズ注入、復号、評価のためのPythonベースのシミュレーションフレームワーク。`simulator/` ディレクトリを参照。

## エラーモデルと状態分離性

See also: [docs/limitations.md](docs/limitations.md)

現実の光システムには、複数のノイズ源とエラー源があります。
このシミュレーターでは、次の要素をモデル化します。

| エラー源         | 説明                      | モデル化           |
| ------------ | ----------------------- | -------------- |
| **ガウスノイズ**   | 測定された光パラメータのランダムな揺らぎ    | 各次元への加法的ガウスノイズ |
| **検出器量子化**   | 有限の検出器分解能が識別可能なレベルを制限する | 床関数 / 天井関数量子化  |
| **チャネルドリフト** | 光特性のゆっくりした系統的変動         | 線形ドリフト項        |
| **スペクトル重なり** | 隣接波長チャネルの漏れ込み           | ガウス型スペクトル交差結合  |
| **偏光エラー**    | 偏光状態の回転や楕円率誤差           | 角度回転ノイズ        |
| **位相不安定性**   | 熱的・機械的ゆらぎによる位相変動        | 位相ノイズ項         |
| **時間ジッター**   | パルス到達時刻やタイムビン割り当ての不確実性  | ジッター分布         |

**復号戦略：**

標準の復号器は、正規化された状態空間における最近傍距離を用います。
受信状態を、定義済みアルファベットの中で最も近い状態へ割り当てます。

**評価指標：**

* アルファベットサイズとノイズレベルに対するシンボルエラー率（SER）
* 混同行列：どの状態がどの状態と混同されやすいか
* 分離マージン：隣接状態間の最小距離

## 最小プロトタイプ・ロードマップ

See also: [docs/roadmap.md](docs/roadmap.md)

### Phase 0: ソフトウェアシミュレーション（現在）

* Pythonで光珠アルファベットを定義する
* 整数値を光珠状態ベクトルとして符号化する
* 設定可能なガウスノイズを注入する
* 最近傍距離で復号する
* シンボルエラー率を測定する
* 混同行列を生成・可視化する
* 冗長符号化戦略をテストする

**目標:**
指定されたノイズレベルにおいて、N状態を目標精度以上で符号化・復号できることを示す。

### Phase 1: 古典光プロトタイプ

* **光源:** RGB LEDアレイまたは波長可変ダイオードレーザー
* **偏光:** 直線偏光子と波長板
* **検出:** カラーセンサー、分光器、多チャネル光検出器
* **復号器:** Pythonベースの最近傍分類器
* **目標アルファベット:** 6〜24識別可能状態
* **指標:** 実験室条件下でシンボルエラー率 < 5%

### Phase 2: 多自由度光プロトタイプ

* 波長 + 偏光 + タイムビン符号化を組み合わせる
* 目標アルファベット：64〜256状態
* キャリブレーション手順とドリフト補正
* 前方誤り訂正の統合
* 再現可能なキャリブレーション・測定ドキュメント

### Phase 3: Qudit互換研究拡張（長期）

* 単一光子源（例：SPDCまたは量子ドット）
* タイムビンまたは周波数ビンqudit符号化
* 経路符号化と干渉計型ゲート
* 量子ゲート操作との互換性評価
* **注意:** このフェーズは決定論的プロトタイプには不要であり、別個の長期研究方向です。

## 光媒体の安定化

See also: [docs/optical-medium-stabilization.md](docs/optical-medium-stabilization.md) | [diagrams/optical-medium-options.md](diagrams/optical-medium-options.md) | [simulator/optical_medium_comparison.py](simulator/optical_medium_comparison.py)

OBQCは、開放空気中の光路を必須としません。
近未来のプロトタイプでは、光珠チャネルを密閉するか、開放空気の隙間をより制御された媒体で置き換えることで安定化できます。

| 媒体               | 近未来の実現性 | 主な利点           | 主なリスク              |
| ---------------- | ------- | -------------- | ------------------ |
| 開放空気             | 非常に容易   | 最も簡単な初期試験      | ほこり、湿度、乱流          |
| 密閉空気 / 窒素        | 容易      | ほこりと湿度を排除      | 温度・圧力ドリフト          |
| 密閉液体（水、シリコーンオイル） | 中程度     | 密閉され屈折率が安定した光路 | 気泡、熱的屈折率ドリフト、位相不安定 |
| アクリル / PMMAブロック  | 容易〜中程度  | 固体・小型・対流なし     | 応力複屈折（偏光DOFには不向き）  |
| 光学ガラス / 石英       | 中程度     | 低応力、最高の光学品質    | 加工コスト              |
| ファイバー / 導波路      | 高度      | 導波・安定・スケーラブル   | 結合の複雑さ             |

重要な考慮点：

* 超純水は、脱気・密閉・温度制御を必要とする密閉液体媒体であり、固体材料ではありません。
* アクリル / PMMAは、波長、強度、空間符号化には適していますが、応力複屈折が大きいため偏光自由度を劣化させます。
* 偏光または位相符号化が必要な場合は、ガラスまたは石英が推奨されます。
* ファイバーと導波路は、スケーラブル実装へ向かう長期的な道です。
* 量子光学拡張では、古典OBQCプロトタイプよりも厳しい損失、散乱、位相コヒーレンス制御が必要です。液体やポリマー媒体が自動的に適するわけではありません。

## ハイブリッド量子支援層

See also: [docs/hybrid-quantum-support-layer.md](docs/hybrid-quantum-support-layer.md) | [simulator/hybrid_quantum_support_energy.py](simulator/hybrid_quantum_support_energy.py) | [simulator/pattern_vs_binary_operation_cost.py](simulator/pattern_vs_binary_operation_cost.py)

OBQCは、超伝導量子プロセッサを置き換えるものでも、極低温冷却を不要にするものでもありません。
超伝導量子ビットは動作にミリケルビン温度を必要とし、光学補助層がそれを変えることはありません。

しかし、OBQCは、ハイブリッド量子システムを取り巻く周辺インフラのための、低エネルギーなパターン認識支援層として探索できます。

多くの近未来量子コンピュータは、量子プロセッサと、室温で動作する大規模な古典インフラを組み合わせたハイブリッドシステムです。
この古典インフラには、制御電子回路、読み出し判別、シンドローム復号、後処理が含まれます。

システムが大規模化するにつれ、この補助インフラは大きく増加する可能性があります。

OBQC的な光パターン認識層は、次を減らせる可能性があります。

* 読み出し信号分類のオーバーヘッド
* 読み出し電子回路と古典復号器の間のデータ移動
* クライオスタットへのI/Oチャネルの高次元多重化
* エラー候補検出における反復的な二値演算
* 特定の古典後処理ワークロードのエネルギーコスト

ただし、その正味利益は条件付きです。
光層が消費するエネルギーが、補助処理で節約されるエネルギーより少ない場合にのみ有用です。
この条件は、トイエネルギーモデル内で明示的かつ反証可能に設定されています。

| 対象           | OBQCの役割          | 限界                    |
| ------------ | ---------------- | --------------------- |
| 超伝導量子システム    | 補助的な読み出し / 前処理層  | 量子ビットの極低温冷却は不要にならない   |
| フォトニック量子システム | より自然な光状態支援層      | 検出器と損失の制約は残る          |
| 古典後処理        | 反復ワークロードのパターン型削減 | オーバーヘッドが節約分より低い場合のみ有用 |
| エラー候補検出      | 反復エラーパターンの高速認識   | デジタル復号器とのベンチマーク検証が必要  |
| I/O多重化       | チャネルごとの高次元光符号化   | 統合と変換のオーバーヘッド評価が必要    |

トイエネルギーモデル（`simulator/hybrid_quantum_support_energy.py`）は、5つのシナリオで正味利益条件を評価します。

* conservative
* moderate
* optimistic
* high_OBQC_overhead
* no_benefit

また、パターンコストモデル（`simulator/pattern_vs_binary_operation_cost.py`）は、どのワークロードとオーバーヘッド条件で、パターン認識が逐次的な二値処理より低コストになるかを特定します。

## 密閉液体型・光珠媒体

See also: [docs/sealed-liquid-optical-bead-medium.md](docs/sealed-liquid-optical-bead-medium.md) | Simulator: [simulator/liquid_medium_noise.py](simulator/liquid_medium_noise.py)

近未来の古典的 Optical Bead Computing プロトタイプでは、光チャネルは開放空気である必要はありません。
密閉された透明液体媒体、すなわち**浸漬安定化光セル**を用いることで、ほこり、空気乱流、湿度変化を排除し、環境安定性を高められる可能性があります。

このモデルでは、光珠状態は、閉じた透明液体セルを通過して伝送されます。
例として、蒸留水、グリセロール水溶液、光学グレードオイルを満たした分光キュベットなどが考えられます。

液体は、機械的に安定した密閉伝搬媒体を提供します。

### 潜在的利点

* ほこりや空中粒子の混入：密閉により排除
* 空気乱流（熱ゆらぎ）：排除され、ゆっくりした液体対流に置き換わる
* 湿度変動：密閉セル内では排除
* 窓ガラスとの屈折率整合：セル境界での反射損失を低減

### 既知の限界

| 限界            | 物理的機構                          | 深刻度                   |
| ------------- | ------------------------------ | --------------------- |
| 吸収損失          | Beer-Lambert減衰：I = I₀ exp(−αL) | 低（5 cm水・可視波長で < 1 dB） |
| 気泡 / 不純物による散乱 | ランダムな振幅低下                      | 低〜中（脱気・濾過で低減可能）       |
| 温度依存の屈折率ドリフト  | 水で dn/dT ≈ −1×10⁻⁴ K⁻¹         | 位相符号化では重大             |
| 波長依存透過率       | 波長ごとに異なるα                      | 中程度、チャネルごとの校正が必要      |
| 長期材料劣化        | 酸化、微生物増殖、アウトガス                 | 低（化学的に安定な媒体と密閉容器を使用）  |
| 温度による位相不安定    | 5 cm光路で0.1 Kあたり約5 radの位相シフト    | 深刻、下記参照               |

### 重要な知見：センチメートル級光路長では位相符号化は非現実的

シミュレーション（`simulator/liquid_medium_noise.py`）では、水中5 cmの光路長において、わずか0.1 Kの温度偏差が約5 radの位相ドリフトを生じることが示されています。
これは、位相符号化された光珠状態を完全に乱すのに十分です。

5 cmで < 0.1 rad の位相安定性を達成するには、2 mK未満の温度制御が必要になります。
これは近未来プロトタイプとしては現実的ではありません。

実用上の意味：

Phase 1およびPhase 2の液体媒体プロトタイプでは、波長符号化と偏光符号化のみを使うべきです。
位相符号化は、能動的な干渉計安定化またはサブミリメートル級の光路長が利用できるまで延期すべきです。

### 液体媒体の比較

| 媒体                | 透過範囲        |       屈折率 | dn/dT (K⁻¹) | 推奨用途           |
| ----------------- | ----------- | --------: | ----------: | -------------- |
| 蒸留水               | 200–1300 nm |     1.333 |   −1.0×10⁻⁴ | Phase 1基準、低コスト |
| グリセロール水           | 200–1200 nm | 1.33–1.47 |   −2.5×10⁻⁴ | 屈折率調整、高いdn/dT  |
| 浸漬オイル             | 400–800 nm  |     1.515 |   −3.0×10⁻⁴ | 屈折率整合窓、短距離     |
| Fluorinert FC-770 | 250–2000 nm |     1.275 |   −1.7×10⁻⁴ | 最広範囲、高コスト      |

このアプローチは、古典光プロトタイプの設計選択です。
巨視的な多光子パルスを扱うものであり、単一光子を扱うものではありません。
量子コヒーレンスや量子優位性を可能にするものではありません。

## SCD-CMOS / LED-CMOS そろばん型パターンアーキテクチャ

See also: [docs/scd-cmos-led-pattern-architecture.md](docs/scd-cmos-led-pattern-architecture.md) | [docs/scd-cmos-led-pattern-architecture_ja.md](docs/scd-cmos-led-pattern-architecture_ja.md) | Diagram: [diagrams/scd-cmos-led-architecture.md](diagrams/scd-cmos-led-architecture.md) | Simulator: [simulator/led_cmos_scd_pattern_demo.py](simulator/led_cmos_scd_pattern_demo.py)

このセクションでは、そろばんからOBQCへの類推をさらに拡張し、SCD電子論理をLED光パターンとCMOSイメージセンサー読み出しへ接続します。
これにより、電子的な10進パターンから Optical Bead Computing へ向かう物理プロトタイプの道筋を作ります。

### 中核アイデア

そろばんの珠には、物理的に2つの状態があります。

* 入っている / 入っていない
* 上がっている / 下がっている
* engaged / disengaged

CMOS配線にも、電気的に2つの状態があります。

* High
* Low

この2つは構造的に同一です。

したがって、そろばんの1桁は、5本のCMOS信号線へ直接写像できます。
これが SCD（Soroban-Coded Decimal）です。

```text
D = (H, L4, L3, L2, L1)

digit = 5*H + count(L1, L2, L3, L4)
```

同じオン / オフ構造はLEDにも拡張できます。
各信号線が1つのLEDを駆動し、CMOSイメージセンサーが空間的な明るさパターンを読み取り、再び数字へ復号します。

### 表現レイヤーのスタック

| レイヤー          | 表現             | 役割            |
| ------------- | -------------- | ------------- |
| そろばん          | 珠の上下           | 物理的な10進パターン   |
| CMOS SCD      | 5本のHigh / Low線 | 電子的な10進パターン   |
| LED / RGB LED | 光の位置、色、強度      | 光パターン生成器      |
| CMOSセンサー      | ピクセルアレイ        | 光パターン読み取り器    |
| OBQC          | 多自由度の光珠状態      | 一般化された光パターン計算 |

### SCD-CMOSとは何か、何ではないか

* SCDは、10進1桁あたり5ビットを使います。BCDは4ビットです。したがって、SCDはBCDよりコンパクトではありません。
* SCDの価値は、記憶サイズ削減ではなく、構造化パターン認識と妥当性チェックにあります。
* 下位4ビットはサーモメーターコードを使います。サーモメーター型でないパターンは構造的に無効です。32通り中22通りは、追加のパリティビットなしでエラー状態として検出できます。
* すべてのエラーを検出できるわけではありません。Hビットが反転すると、有効な数字が別の有効な数字へサイレントに変わる可能性があります。例：0→5、1→6。SCDは構造的妥当性コードであり、完全な誤り訂正コードではありません。

### LED-CMOS拡張

LEDディスプレイとCMOSセンサーは、純粋な二値電子回路にはない光の自由度を追加します。

* 色 / 波長
* 明るさ / 強度
* 空間的位置
* 時間的パルスパターン

これにより、SCDはOBQCへ接続されます。

* SCDは、5本の二値線にまたがる構造化された電子珠パターンを使う。
* LED-CMOSは、そのパターンを光空間へ拡張する。
* OBQCは、珠パターンを波長、偏光、位相、タイムビン、空間モードなどの多次元光状態へ一般化する。

### トイシミュレーター

[simulator/led_cmos_scd_pattern_demo.py](simulator/led_cmos_scd_pattern_demo.py) は、次を実演します。

* SCD数字符号化
* LED明るさパターンへの写像
* ガウス型明るさノイズの注入
* CMOS風しきい値読み出し
* 正解 / 検出エラー / サイレントエラーの分類

これは、明るさノイズのトイモデルです。
物理光学シミュレーションではありません。

## 電子拡張：Soroban-Coded Decimal Logic

See also: [docs/electronic-extension-soroban-decimal.md](docs/electronic-extension-soroban-decimal.md) | Simulator: [simulator/soroban_decimal.py](simulator/soroban_decimal.py)

そろばんの構造論理は、フォトニックシステムに限定されません。

これは、電子回路設計へ直接写像できます。
その形が **Soroban-Coded Decimal（SCD）** です。

SCDは、デジタルハードウェア上でそろばんの1軸の物理構造を反映する、5ビット制約付き10進セルです。

### SCDセル定義

```text
D = (H, L4, L3, L2, L1)

digit = 5 * H + count(L4, L3, L2, L1)
```

ここで、Hは天珠（値5）、L1〜L4は地珠（各値1）です。
地珠は、隙間なく順に有効化される**サーモメーターコード**で符号化されます。

### 符号化表

| Decimal |  H | L4 | L3 | L2 | L1 | 5-bit code |
| ------: | -: | -: | -: | -: | -: | ---------- |
|       0 |  0 |  0 |  0 |  0 |  0 | `0 0000`   |
|       1 |  0 |  0 |  0 |  0 |  1 | `0 0001`   |
|       2 |  0 |  0 |  0 |  1 |  1 | `0 0011`   |
|       3 |  0 |  0 |  1 |  1 |  1 | `0 0111`   |
|       4 |  0 |  1 |  1 |  1 |  1 | `0 1111`   |
|       5 |  1 |  0 |  0 |  0 |  0 | `1 0000`   |
|       6 |  1 |  0 |  0 |  0 |  1 | `1 0001`   |
|       7 |  1 |  0 |  0 |  1 |  1 | `1 0011`   |
|       8 |  1 |  0 |  1 |  1 |  1 | `1 0111`   |
|       9 |  1 |  1 |  1 |  1 |  1 | `1 1111`   |

5ビットで表現可能な32状態のうち、有効状態は10通りです。
残りの22状態は、定義上、検出可能なエラー状態です。
サーモメーター型でない下位パターンは不正であり、追加のパリティビットなしでフラグできます。

### 桁上がり・借りの構造

インクリメントとデクリメントは、そろばんの機械的論理に直接従います。

* **4 + 1:** 下位珠が4個に到達し、H = 0 → H = 1にし、下位珠をクリアする（そろばんの「5の補数」型繰り上げ）
* **9 + 1:** H = 1、下位珠が満杯 → セルを0へ戻し、次桁へ繰り上げ
* **5 − 1:** H = 1、下位 = 0 → Hをクリアし、下位を1111にする（5グループからの借り）
* **0 − 1:** セル = 0、下位 = 0 → 9にし、次桁から借りる

### 光レイヤーとの関係

SCDは、光珠状態の電子的アナログです。

| 領域      | 情報単位      | 構成要素1        | 構成要素2             | 有効状態         |
| ------- | --------- | ------------ | ----------------- | ------------ |
| 物理そろばん  | 軸上の珠配置    | 天珠 H         | 地珠（サーモメーター）       | 10           |
| SCD（電子） | 5ビット10進セル | Hビット         | L1〜L4ビット（サーモメーター） | 32中10        |
| OBC（光）  | 光珠状態 B    | λ, P, φ, ... | 多自由度符号化           | アルファベット定義による |

三者はすべて、同じ構造原理を共有します。

> 値は、線形な二値列ではなく、多構成要素パターンである。

この対応により、認知、電子、フォトニックの各領域を横断する統一的な概念フレームワークが可能になります。

### SCDの位置づけ

SCDは、記憶密度でBCDを置き換えるためのものではありません。
その価値は、構造的・教育的な点にあります。
つまり、そろばん演算を回路論理上で直接観察可能にすることです。

## OBQCが主張しないこと

信頼性のため、次の明確化が重要です。

* **OBQCは、完成した汎用量子コンピュータであるとは主張しません。**
  量子ゲート、誤り訂正済み量子ビット、量子優位性は実証されていません。

* **OBQCは、汎用計算でCPU、GPU、古典コンピュータを上回るとは主張しません。**
  この段階で比較ベンチマークは存在しません。

* **OBQCは、理論上のすべての光状態を信頼性高く識別できることの証明ではありません。**
  理論状態数は上限です。実用アルファベットサイズはより小さく、実験的に決定する必要があります。

* **OBQCは、単なるフォトニック多重化方式ではありません。**
  WDM、PDM、OFDMなどの光多重化は複数自由度を帯域のために使いますが、OBQCはそれらを、そろばん型パターン論理に着想を得た多値アルファベットとして構造的に使います。

* **OBQCは、提案フレームワークです。**
  そろばん着想型の多値光符号化とパターン型計算を、明確な実験ロードマップと反証可能な主張に基づいて探索するためのものです。

## 反証可能な主張

次の主張は、具体的で測定可能かつ検証可能です。

1. **状態符号化精度:**
   定義されたN状態の光珠アルファベットについて、指定ノイズレベルで、符号化と最近傍復号により目標しきい値未満のシンボルエラー率（例：< 5%）を達成できるか。

2. **SERスケーリング:**
   固定ノイズ条件下で、アルファベットサイズNが増えるとシンボルエラー率はどのように増加するか。

3. **自由度の頑健性:**
   波長、偏光、位相など、どの光自由度が現実的ノイズ条件下で最も高い分離マージンを持つか。

4. **冗長性による改善:**
   多数決や反復符号化などの冗長符号を適用すると、シンボルエラー率は測定可能に低下するか。

5. **パターン符号化による圧縮:**
   選択された構造化データ型について、光珠パターン符号化は二値符号化より少ないシンボル数で値を表せるか。

これらの主張は、ハードウェアを構築する前に、まずソフトウェアシミュレーション（Phase 0）で検証できます。

## 限界

See also: [docs/limitations.md](docs/limitations.md)

| 限界                  | 説明                                                  |
| ------------------- | --------------------------------------------------- |
| **ノイズ**             | すべての光システムはショットノイズ、熱雑音、増幅器ノイズの影響を受け、識別可能な状態数が減る      |
| **クロストーク**          | 波長、位相、偏光次元の隣接状態が互いに漏れ込む可能性がある                       |
| **光学アライメント**        | 位相・偏光測定には精密なアライメントが必要で、機械的摂動がエラーを引き起こす              |
| **検出器の限界**          | 検出器の分解能、ダイナミックレンジ、サンプリングレートが達成可能な状態分離性を制限する         |
| **スケーラビリティ**        | 自由度を追加すると理論上は状態数が乗法的に増えるが、実際には新たなクロストーク源も導入される      |
| **キャリブレーションドリフト**   | 光システムは熱膨張、機械緩和、部品劣化により時間とともにドリフトする                  |
| **エネルギーコスト**        | レーザー光源、変調器、量子拡張用の極低温検出器は大きな電力を消費する                  |
| **ハードウェア複雑性**       | 多自由度光システムには、精密で高価かつ環境に敏感な部品が必要                      |
| **理論状態数と使用可能状態数の差** | 理論状態積 N = n_λ × n_P × ... は、常に実用アルファベット数を上回る        |
| **実証ハードウェアなし**      | 2026-06-03時点では物理プロトタイプは存在せず、結果はすべてソフトウェアシミュレーションによる |
